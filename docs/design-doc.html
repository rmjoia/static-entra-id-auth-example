Design & Implementation Reference

Static website with Microsoft Entra ID authentication on Azure Static Web Apps

Two complementary patterns for hosting a public-facing site whose protected pages require sign-in: a Free-plan pattern with per-user invitations, and a Standard-plan pattern with group-based access via a custom app registration and a role-assignment function.

1. Executive summary

This document describes a reference implementation for hosting a static website on Azure Static Web Apps (SWA) where some pages are public and others require authenticated access through Microsoft Entra ID. The pattern is intended for use cases such as board reports, investor updates, partner-only documents, or any site that needs a public marketing surface paired with a private members-only area, without operating a traditional web server.

Two parallel implementations are provided in the same source repository:

  • Free plan Uses the platform's pre-configured Microsoft Entra ID provider. Access to protected pages is granted via per-user invitation links. Suitable for fixed audiences of up to 25 users. No app registration is required; the authentication provider is multi-tenant by default.
  • Standard plan Uses a custom Microsoft Entra ID app registration scoped to a specific tenant, with role assignment driven by membership in an Entra ID security group. Authorization is computed by a serverless function at sign-in. Suitable for audiences that fluctuate, exceed 25 members, or are managed by existing identity-governance processes.

Both implementations share a common front-end (Bootstrap 5 + Chart.js, Microsoft-branded "Contoso annual review" demo). They differ only in staticwebapp.config.json and, for the Standard variant, the presence of a GetRoles Azure Function under the /api path.

The reference is deployed via GitHub Actions workflows generated by the Azure portal. Each Static Web App instance owns its own workflow file; path-scoped triggers ensure that pushes affecting only one tier do not redeploy the other.

Headline recommendations

  1. For demo and small fixed audiences, deploy as Free-tier. No additional security work is required for the platform itself.
  2. For audiences managed through Entra groups, deploy as Standard-tier. Combined with Conditional Access (MFA, compliant device) on the underlying app registration, this is suitable for sensitive but not regulated content.
  3. Before exposing real confidential data: replace the client secret with a Key Vault-stored certificate fetched by the SWA's managed identity, and enable Conditional Access policies on the app registration covering MFA and (where applicable) device compliance.
  4. For regulated, highly sensitive, or partner-only content: pair Standard-tier with either a private endpoint (intranet-only) or Azure Front Door Premium with WAF (public-internet but filtered). Both options are documented in this paper.

3.1 Trust posture

  • Authentication. The pre-configured provider is multi-tenant. Any holder of a Microsoft account — personal or work, in any tenant — can complete the sign-in handshake. This means authentication is not, by itself, an access barrier.
  • Authorization. Protected routes (/report.html, /strategy.html) carry an allowedRoles: ["boardmember"] rule. The custom role boardmember is granted only by invitation, generated from the SWA's Role management blade.
  • Capacity. The invitation system is capped at 25 users per app on both Free and Standard plans. To exceed this, the Standard-tier pattern is required.
  • Revocation. Removing a user from Role management invalidates their permissions on the next sign-in; the SWA's session cookie continues to honour previously assigned roles until expiry.

3.2 What this pattern is good for

  • Fixed, small audiences (board of directors, investor short-list, partner pilot).
  • Reference architectures and demonstrations.
  • Content whose value does not justify the operational overhead of an app registration.

3.3 What this pattern is not suitable for

  • Audiences larger than 25 members.
  • Audiences whose composition changes frequently (joiners/leavers, conditional access requirements).
  • Use cases requiring restriction to a specific Entra tenant at the sign-in level.
  • Use cases requiring an SLA on uptime.

4.2 Required app registration settings

  • Supported account types: Single tenant. Restricts sign-in to the org's own users.
  • Redirect URI: Web platform — https://<swa-host>.azurestaticapps.net/.auth/login/aad/callback.
  • Implicit grant and hybrid flows: ID tokens enabled. Without this, sign-in fails with AADSTS700054: response_type 'id_token' is not enabled for the application.
  • Optional claims (ID token): groups claim added. Type: Security groups (or All groups, depending on operational preference).
  • API permissions: default Microsoft Graph — User.Read (delegated) is sufficient.
  • Credentials: client secret in Phase 1; certificate stored in Key Vault in Phase 2 hardening.

4.3 The GetRoles function

The function is Node.js, Azure Functions v4 programming model. It receives the platform's POST at every sign-in, with the user's identity and claims in the body. It is internal-only by virtue of being declared as rolesSource; the platform automatically blocks external HTTP requests to it.

// standard-tier/api/src/functions/GetRoles.js (excerpt)
app.http("GetRoles", {
  methods: ["POST"],
  authLevel: "anonymous",  // platform-only invocation
  handler: async (request, context) => {
    const body = await request.json();
    const groupIds = new Set(
      (body.claims || [])
        .filter(c => c.typ === "groups")
        .map(c => c.val)
    );
    const boardGroupId = process.env.BOARD_GROUP_ID;
    const roles = [];
    if (boardGroupId && groupIds.has(boardGroupId)) {
      roles.push("boardmember");
    }
    return { status: 200, jsonBody: { roles } };
  }
});
Group claim overage. If a user belongs to more than ~150 Entra groups, Microsoft omits the groups claim from the token and emits _claim_names / _claim_sources pointing to Microsoft Graph instead. The reference implementation does not include the Graph fallback. For small teams this is irrelevant; for large organisations, see the Microsoft Learn "Assign roles via Graph" tutorial (§13).

5.1 Source of truth for runtime authorization

The Static Web App platform's /.auth/me endpoint returns the active clientPrincipal, including the userRoles applied to the current session. This is the authoritative value used by the platform for route evaluation. The Role management blade and the function's return value determine what is written to a new cookie; once a cookie is issued, its claims hold until the cookie expires or is invalidated.

Operational consequence: a change in group membership (Standard) or invitation roles (Free) does not affect existing sessions. Users must sign out and back in to receive an updated role set. This is documented in Operations (§11).

5.2 Edge propagation

Role changes propagate across the Static Web Apps edge network with a typical latency of several minutes (documented by Microsoft as "a few minutes"). During the propagation window, different geographically distributed edge nodes may return different userRoles for the same user on parallel requests. The behaviour stabilises once propagation completes and is irreversible (the new value, once observed, persists).

7.4 What changes in the SWA when CA is added?

Nothing. The website's configuration, code, and behaviour are unchanged. A sign-in that fails CA never produces a token; the SWA's authentication endpoint simply never sees a success and the browser remains on a Microsoft-hosted block page. The website's /403.html is shown only when sign-in succeeds but the resulting roles are insufficient — this remains the same after CA is introduced.

8.1 Workflow generation

Each workflow file is generated by the Azure portal at SWA creation time. The portal injects a deployment token as a GitHub repository secret named AZURE_STATIC_WEB_APPS_API_TOKEN_<NAME>. The workflow uses Azure/static-web-apps-deploy@v1 to upload the build output. Path filters were added manually after the portal-generated baseline.

8.2 Operational characteristics

  • Build time: 1–3 minutes for an unchanged-dependencies deployment. The static assets are tiny and the API folder has no native dependencies.
  • Deployment model: blue-green via SWA's staging environments. Pull requests automatically create a preview environment per the workflow's close_pull_request_job; merging to main rolls the preview to production.
  • Rollback: revert the offending commit on main. The next push redeploys the previous content. SWA does not maintain historical builds.
  • Branch protection: not enforced in the reference; production deployments should require a pull-request review and a passing status check before merge to main.

9. Hardening roadmap

The reference implementation is a starting point. The phases below describe a progression of incremental security improvements, each with a clear marginal cost and benefit. Phase 1 is the current state; Phases 2–5 are independent — you can adopt any subset.

Phase 1 Reference deployment (as-is)

Standard-tier SWA, custom Entra app with client secret, GetRoles function, group-based access. No Conditional Access. Public DNS resolves the SWA URL.

  • Authentication: custom Entra ID, single-tenant.
  • Credential: client secret in SWA application settings.
  • Authorization: boardmember role assigned by GetRoles based on BOARD_GROUP_ID.
  • Network: public internet; basic platform DDoS protection only.

Suitable for: reference / demo with non-sensitive content.

Phase 2 Replace client secret with Key Vault certificate

Eliminates the plaintext credential from SWA application settings. Cert is generated and held inside Azure Key Vault; the SWA accesses it via a system-assigned managed identity.

  • Generate certificate in Key Vault. Private key never leaves the vault except into SWA runtime memory.
  • Upload public key (.cer) to the Entra app registration.
  • Grant the SWA's managed identity Get/List on certificates and secrets in Key Vault.
  • Replace clientSecretSettingName with clientSecretCertificateKeyVaultReference in staticwebapp.config.json.
  • Remove the AZURE_CLIENT_SECRET application setting.

Effort: 30–60 minutes. Recurring cost: negligible (Key Vault operations are below 100 per month). Suitable for: production with any meaningful confidentiality.

Phase 3 Conditional Access policies on the app registration

Require MFA at minimum. Optionally require a compliant device. Optionally restrict to trusted locations.

  • Create CA policy targeting the SWA's app registration and the board-members group.
  • Grant: Require MFA. Optionally + compliant device.
  • Roll out in Report-only mode for several days; verify in Insights; then switch to On.

Effort: 1–3 days including verification. Recurring cost: Entra ID P1 per user (typically already owned). Intune per user if device compliance is enforced. Suitable for: any deployment with internal organisational users.

Phase 4 Network restrictions

Choose one (or combine):

  • IP restrictions (Standard plan, free) — allow-list up to 25 IP ranges. Blocks all other traffic at the platform.
  • Private endpoint (Standard plan, free, but requires a VNet) — SWA URL only resolves from within the connected VNet. Pair with VPN or ExpressRoute for remote users.
  • Azure Front Door Premium with WAF — keeps public DNS but filters traffic against the OWASP Core Rule Set, applies rate limiting, geo-blocking, bot protection. Material monthly cost (see §10).

Effort: half a day to several days depending on option. Recurring cost: zero (IP restrictions / private endpoint) to several hundred dollars per month (AFD Premium). Suitable for: regulated content or content with substantial value.

Phase 5 Operational maturity

Monitoring, alerting, repo hygiene, and incident-response readiness.

  • Application Insights integration for sign-in telemetry and function logs.
  • Alerts on anomalous sign-in volume, function errors, or role assignment patterns.
  • GitHub: branch protection on main, secret scanning + push protection, periodic rotation of SWA deployment tokens.
  • Entra: enable group-membership audit alerts via Azure Monitor; consider Privileged Identity Management (PIM) for group ownership.

Effort: 1–2 days initial, ongoing low effort. Recurring cost: negligible. Suitable for: any deployment expected to last longer than 6 months.

11.3 Credential rotation (Standard, client-secret pattern)

  1. Entra portal — App registrations — your app — Certificates & secrets — New client secret. Generate with a new expiry; copy the Value.
  2. SWA — Environment variables — update AZURE_CLIENT_SECRET with the new Value.
  3. Verify sign-in works (incognito test).
  4. Once verified, return to Entra and delete the old client secret entry.

11.4 Credential rotation (Standard, certificate pattern)

  1. Key Vault — Certificates — your cert — New version (if auto-rotation is not configured).
  2. Download the new .cer from Key Vault.
  3. Entra portal — App registrations — Certificates & secrets — Certificates — Upload the new .cer.
  4. Verify sign-in works (incognito test). Because clientSecretCertificateThumbprint is "*", the runtime auto-picks the latest version.
  5. Optionally delete the previous cert version after the verification window.

11.5 Monitoring

  • Application Insights connected to the Standard SWA captures GetRoles invocations via context.log(...). Use this to detect anomalies (unusual user volume, unexpected groupCount values, etc.).
  • Entra sign-in logs (Entra portal — Sign-in logs) show all sign-in attempts including CA evaluations, MFA outcomes, and policy hits.
  • Configure alerts in Azure Monitor for: function error rate exceeding threshold, sign-in failure rate spikes, unusual geographic sign-in patterns.

Tenant admin compromisedOut of scope for this design; follow organisational IR procedures. Rotate all app credentials and review CA policies once contained.
Group owner compromisedAudit group membership immediately; revoke any unauthorised members; rotate group owner; review sign-in logs for the affected window.
SWA misconfigured (e.g., wrong tenant ID in config), no users can sign inRevert the offending commit on main; next deploy restores previous state. SWA has no built-in rollback.
